SocialMediaReader V0 开发文档

V0 目标是构建一个面向个人使用的 AI 工作流调研系统：持续收集 GitHub 和专业技术内容， 通过 Dify 完成摘要、分类、对比与周报生成，最终在个人网站中展示。

版本：V0 主题：AI 工作流搭建 部署：Ubuntu 2 核 4G 日期：2026-05-13
核心目标 每天低频采集，每周自动生成调研报告，并支持周内查看最新进展。
核心来源 GitHub 项目发现和技术方案对比优先，CSDN、掘金等中文内容作为补充。
核心边界 暂不做视频画面理解、社交媒体深度采集、多用户系统和本地大模型。

1. 初始系统设计

系统采用 Dify + 自研采集器 + 个人网站 的组合。Dify 负责 AI 工作流， 自研服务负责数据采集、去重、调度、持久化和网站 API。这样可以避免从零搭建 Agent 平台， 同时保留对数据来源和报告展示的控制力。

专业信息源 GitHub、官方文档、技术博客、CSDN、掘金。
D1 SourceQuery
Collector 按主题关键词采集候选项目、文章、Release 和基础元数据。
D2 RawItem
Processor 清洗正文、URL 去重、内容指纹、基础质量过滤。
D3 AnalysisInput
Dify 内容分析 输出相关性、质量评分、摘要、标签和技术路线。
D2/D4
D4 AnalysisResult
Scheduler 每日采集、每日分析、每周报告，也支持手动刷新。

SQLite Storage 保存 topics、sources、github_repos、analyses、reports，作为系统事实来源。
D5 ReportInput
Dify 周报生成 基于一周内高质量分析结果生成中英文报告和引用列表。
D6 ReportResult
API Service 提供主题、仪表盘、来源、Repo 和报告查询接口。

架构图数据格式说明

代称 流向 数据内容
D0 JobCommand Scheduler -> Collector / Processor / Report Job 任务类型、topic_id、时间范围、触发方式、重试次数、最大采集数量。
D1 SourceQuery Collector -> 专业信息源 关键词、source_type、排序方式、时间范围、分页参数、GitHub topic、language 过滤条件。
D2 RawItem Collector -> Processor / Storage title、url、author、published_at、collected_at、source_type、description、raw_text、GitHub 元数据。
D3 AnalysisInput Processor -> Dify 内容分析 topic、title、url、description、readme 摘要、stars、forks、language、license、last_commit_at。
D4 AnalysisResult Dify 内容分析 -> Storage relevance_score、quality_score、summary_zh、summary_en、tags、technical_route、pros、cons。
D5 ReportInput Storage -> Dify 周报生成 topic、period_start、period_end、高分项目列表、技术路线聚合、摘要列表、引用来源。
D6 ReportResult Dify 周报生成 -> Storage title、report_zh、report_en、cited_sources、生成时间、报告周期。
D7 ViewModel API Service -> Personal Website 本周新增项目、top_routes、高分内容、notable_changes、周报列表、当前进展摘要。

V0 的系统重心不是“全网抓取”，而是建立一个稳定的信息处理闭环： 采集 -> 清洗 -> 分析 -> 入库 -> 报告 -> 展示。

2. 初始功能设计

V0 必做功能

  • 创建和维护研究主题，初始主题为 AI 工作流搭建。
  • 按关键词从 GitHub 搜索相关开源项目。
  • 抓取 GitHub Repo Detail，包括 stars、forks、license、language、topics、README、release 信息。
  • 对采集内容做 URL 去重、内容指纹去重和基础质量过滤。
  • 调用 Dify 对项目做相关性评分、质量评分、摘要、标签和技术路线归类。
  • 每周自动生成中英文调研报告。
  • 在个人网站展示本周新增项目、技术路线分布、高分项目、周报历史和引用链接。
  • 支持周内查询当前最新调研进展。

V0 暂不做功能

  • 小红书、B站、知乎等社交媒体采集。
  • 视频画面理解和音视频转写。
  • 复杂 Agent 系统开发。
  • 多用户权限、计费、公开分享和团队协作。
  • 本地大模型推理和复杂向量数据库。

3. 模块划分

模块 职责 V0 范围
collector 外部数据采集。 GitHub Search、Repo Detail、Release，后续扩展 RSS、CSDN、掘金。
processor 清洗、去重、评分和调用 Dify。 URL 去重、fingerprint、基础质量评分、内容分析请求。
scheduler 定时执行采集、分析和报告任务。 每日采集、每日分析、每周报告、手动刷新。
storage 数据持久化。 SQLite 存储主题、来源、Repo 元数据、分析结果和报告。
dify_client 封装 Dify API 调用。 内容分析工作流、周报生成工作流。
api 为个人网站提供接口。 主题、仪表盘、报告、来源、Repo 查询。
website 个人网站页面展示。 研究方向、本周动态、周报历史、即时进展。

4. 数据库核心

V0 使用 SQLite。它的优势是部署简单、资源占用低、备份方便，适合 2 核 4G 的个人服务器。 等内容量和查询复杂度上来后，再迁移到 PostgreSQL。

7. 部署方式

推荐使用 Docker Compose 部署。开发期可以先在本地 WSL 中跑通，再通过 Git 同步代码和配置模板到服务器。 生产环境在服务器上配置独立的 .env、Dify Key、GitHub Token 和数据卷。 

docker-compose
- dify
- social-media-reader-api
- scheduler
- sqlite volume
- nginx

本地到服务器同步策略

  • 同步：采集器代码、API 代码、页面模板、Docker 配置、Prompt、Dify 工作流导出文件、部署脚本。
  • 不同步：.env 明文密钥、Docker volumes、Cookie、API Key、生产数据库、大量采集原文。

2 核 4G 条件下不建议本地部署大模型，不建议常驻浏览器自动化，也不建议高频抓取。 LLM 统一使用外部 API，采集任务低频执行。

8. 开发里程碑

  1. M1：GitHub 采集可用。 实现主题关键词、GitHub repo 搜索、repo 元数据入库。
  2. M2：Dify 内容分析。 接入 Dify 工作流，把 repo 转成摘要、标签、评分和技术路线。
  3. M3：个人网站首页。 展示本周新增 repo、评分、摘要、技术路线和链接。
  4. M4：周报生成。 每周自动生成中英文报告并入库。
  5. M5：即时进展查询。 展示本周当前调研情况，支持手动刷新。
  6. M6：扩展内容源。 加入官方博客 RSS、CSDN、掘金等补充来源。

9. 关键技术决策

决策点 选择 理由
AI 工作流平台 Dify 适合做工作流、知识处理和报告生成，减少自研 Agent 平台成本。
后端 Python FastAPI 适合采集、调度、API 和 Dify 集成，生态成熟。
调度 APScheduler V0 单机低频任务足够使用，不需要复杂队列。
数据库 SQLite 轻量、易备份、部署成本低，符合个人项目和 2 核 4G 约束。
前端 FastAPI + Jinja2/HTMX 起步 先保证报告和数据闭环，减少前后端分离的初始复杂度。
LLM 外部 API 服务器资源有限，不适合本地跑大模型。
内容源 GitHub 优先，中文技术社区补充 GitHub 更结构化、可信度更高，适合作为专业调研主来源。
向量检索 V0 暂缓 早期内容规模可通过结构化摘要和标签支撑周报，避免过早复杂化。
工程原则： 先建立稳定闭环，再扩展来源；先做结构化报告，再做自由问答；先用低复杂度方案跑起来，再根据真实瓶颈升级。
